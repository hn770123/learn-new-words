# エンジニアが雰囲気で使ってるIT用語を正しく学ぶ本の企画案(改訂版)

## 書籍コンセプト

**タイトル案:**

- 「それ、本当に合ってます? エンジニアの”なんとなく”IT用語辞典」
- 「プルリク出す前に読む本〜雰囲気で使ってたあの言葉〜」
- 「技術書には載ってない、現場のIT用語のモヤモヤ全部解消」

**ターゲット読者:**

- 実務経験1〜5年のエンジニア
- 資格は持ってるけど現場用語に戸惑う人
- 「分かったフリ」で乗り切ってきた中堅層

-----

## 章立て構成

### 第1章:「Gitで雰囲気で使ってる」系

**特徴:** 毎日使うのに実は曖昧な用語

- **マージとリベース** → 「結果同じじゃね?」からの深い違い
- **フェッチとプル** → 混同しがちな兄弟
- **チェリーピック** → さくらんぼ摘み?コミット摘み?
- **スカッシュ** → 押しつぶすって何を?
- **ステージング** → add したら staging area に…って?
- **HEAD、origin、upstream** → 誰が誰やら問題
- **fast-forward** → 早送り?マージの一種?

**ウケる仕掛け:** 「Git操作、本当は何が起きてるの?」図解

### 第2章:「開発フローで雰囲気で使ってる」系

**特徴:** スクラムとかアジャイルとか

- **スプリント** → なぜ短距離走?
- **バックログ** → 溜まってる仕事…だけ?
- **リファクタリング** → リライトとの違いは?
- **テクニカルデット** → 技術的負債、具体的には?
- **ポストモーテム** → 死後?振り返り?
- **インクリメント** → 増分?成果物?
- **ベロシティ** → 速度を測る意味とは

**ウケる仕掛け:** 「先輩の使い方、実は間違ってた」事例集

### 第3章:「ビジネス会議で雰囲気で使ってる」系

**特徴:** カタカナが飛び交う会議室でエンジニアは死ぬ

ビジネス会議に呼ばれたエンジニアが最初に学ぶこと—それは「日本語で話せ」と心の中で叫んでも無駄だということだ。ROI、KPI、ステークホルダー...横文字のオンパレードに耐えながら、あなたは静かにSlackで「今の会議、結局何が決まったんですか?」と先輩に聞く。でも先輩も実は分かってない。安心してほしい、みんな雰囲気で使ってる。

- **ROI(投資対効果)** → 「このリファクタリングのROIは?」と聞かれて固まるやつ
  - 「バグが減る」は効果として認められないらしい。数字で言えって?知らんがな
  - 技術的負債の利息は財務諸表に載らないから存在しないことになっている(ビジネス側の認識)
  - エンジニア「保守性が上がります」→ 経営層「で、いくら儲かるの?」→ エンジニア「...」

- **KPI、KGI、OKR** → 全部目標っぽいアルファベット3文字の呪文
  - KPI: 毎週変わる指標。先週は「デプロイ回数」が重要だったのに今週は「障害件数」らしい
  - KGI: KPIの親分。でもKPIとの違いを説明できる人は会議室に3人しかいない(うち2人は嘘ついてる)
  - OKR: シリコンバレー発の目標管理手法。導入した結果、目標設定の会議が増えただけだった
  - 「目標」って日本語で言えばいいのに、なぜかアルファベットにすると説得力が増す不思議

- **ステークホルダー** → 「関係者」って言えばいいのに、なぜかカタカナ
  - ステーク(杭)を持ってる人...ではなく、利害関係者のこと
  - 会議の度に「ステークホルダーは誰?」と聞かれるが、結局「全員」という結論に落ち着く
  - 本当のステークホルダー: お前んとこのプロダクトマネージャーだよ、もうそいつに聞いて決めてくれ

- **アライアンス** → RPGの同盟システムじゃないぞ
  - 「戦略的アライアンス」と言われても、ただの業務提携だったりする
  - パートナーシップとの違い?そんなもん気分だ。カッコよく聞こえる方を使え
  - エンジニア的には「API連携先が増えた」くらいの認識でOK

- **リソース** → 人間をリソース呼ばわりする文化、好きじゃない
  - 会議では「リソース不足」と言うが、Slackでは「人が足りない」と言う。同じ意味
  - サーバーリソース、人的リソース、時間リソース...なんでもリソース
  - 「リソースを最適化」= 誰かの残業が増える、の婉曲表現

- **フィージビリティスタディ** → 「できるかどうか調べる」を壮大に言う技術
  - FS: 2週間かけて「無理です」って結論を出すための儀式
  - エンジニア「3日で調査できます」→ PM「いや、ちゃんとFSとして2週間かけて」→ 調査3日+報告書作成に11日
  - 結論: できる、できない、よりも「報告書の厚さ」が重要らしい

- **マイルストーン** → 昔、道路に置かれてた距離を示す石。今はガントチャートの締切日
  - 「次のマイルストーンは?」→「来週の金曜日です」→「じゃあそれに間に合わせて」→ 無理です
  - マイルストーンは踏み台じゃない。でも現実では踏み台にして次のマイルストーンへ急ぐ
  - 未達成のマイルストーンは「リスケ」という魔法で消え、新しいマイルストーンが生まれる

- **ボトムアップ vs トップダウン** → 民主主義か独裁か、それが問題だ
  - ボトムアップ: 現場の意見を吸い上げる(建前)。実際は会議が増えて何も決まらない
  - トップダウン: 経営層が決める(本音)。現場の反発を「ボトムアップで集めた意見」で緩和する
  - 理想: ボトムアップ、現実: トップダウン、会議での説明: 「ボトムアップとトップダウンのハイブリッド」

- **コンセンサス** → 全員一致を目指して、結局誰も責任取らない仕組み
  - 「コンセンサスを取る」= 会議を3回やって、最後は「じゃあこれで」と誰かが諦める
  - 多数決と違う点: 反対者を説得するフリをする時間が長い
  - エンジニア界の真実: コードレビューで2人がLGTM出したらコンセンサス成立

- **タスクフォース** → 特別感を出したいときに使う言葉
  - 普通のチーム: プロジェクトチーム、カッコいいチーム: タスクフォース
  - 緊急性を演出する効果がある。でも実態は「いつものメンバー+1人」
  - 「障害対応タスクフォース」← カッコいい、「障害対応チーム」← 普通、「障害当番」← 現実

- **エスカレーション** → 上司に丸投げする行為を丁寧に言う技術
  - 「エスカレお願いします」は「もう俺には無理、上の人が決めて」の意
  - エスカレーションパス: 誰に丸投げするかの順番を定めたもの
  - 三段階エスカレーション: あなた→上司→部長→役員、最後は誰も決められず現場に戻ってくる

- **グロースハック** → マーケティング用語がエンジニア界に殴り込みしてきたやつ
  - 「技術で成長させる」カッコいい響きだが、実態は「A/Bテストしまくる」
  - エンジニア「パフォーマンス改善します」→ PM「それグロースハックになる?」→ エンジニア「...なる、かも?」
  - Hack(ハック)と付けば何でも許される風潮。次は「デバッグハック」とか出てくるぞ

- **ピボット** → 失敗を認めたくないときの言い換え言葉
  - 「方向転換」じゃなく「ピボット」と言うと、なぜか戦略的に聞こえる
  - 仕様変更: エンジニアが頑張る、ピボット: 経営判断だから仕方ない、という空気の違い
  - 3回目のピボット = もう何作ってるか誰も分かってない

- **MVP(Minimum Viable Product)** → 「最低限動くやつ」を横文字にしただけ
  - Product(製品)という割に、実態はプロトタイプ未満のことも多い
  - 「MVPだから雑でいいよね」→ リリース後→「なんでこんなバグだらけなの?」→ だからMVPだって...
  - Minimum(最小限)の定義が、エンジニアとビジネス側で10倍くらい違う

- **PMF(Product Market Fit)** → 市場に受け入れられてるかの指標。エンジニアには関係ない...と思いきや
  - 「PMF達成!」→ エンジニア「じゃあリファクタリングできますね」→「いや、今は成長フェーズだから機能開発優先で」
  - PMF前: 速く作れ、PMF後: もっと速く作れ。技術的負債は永遠に返済されない
  - エンジニアが知るべきこと: PMFは言い訳に使える。「PMF前だから多少の障害は...」

- **バリュープロポジション** → 「ウチの製品のいいところ」を小難しく言う技術
  - 機能一覧を並べると「それはバリュープロポジションじゃない」と怒られる
  - じゃあ何なのか説明してくれよ...と思うが、説明する側も実は分かってない
  - エンジニア的解釈: README.mdの「Features」セクションに書いてあることを、もっとふんわり言う

- **バーンダウンチャート** → 残タスクを可視化して、チームを絶望させるグラフ
  - 理想線: 右肩下がりの美しい直線、現実線: ジグザグの階段、最後に垂直落下
  - スプリント終盤の垂直落下 = 徹夜で帳尻を合わせた証拠
  - 燃え尽きる(Burn down)のはグラフじゃなくて、エンジニアの心

- **ウォーターフォール** → 「古い」「時代遅れ」の代名詞として使われがち
  - でも実際は、多くの現場が「アジャイルと言いながらウォーターフォール」
  - 滝は一方通行。戻れない。手戻り発生時の絶望感は異常
  - 「ウチはアジャイルだから」と言いつつ、要件定義→設計→実装→テストを順にやってる現場、見てるぞ

- **コアコンピタンス** → 「ウチの強み」をカタカナにしただけ。それ以上でも以下でもない
  - 会議で「コアコンピタンスは何か?」と聞かれ、30分議論して結局決まらない
  - エンジニア「技術力です」→ ビジネス「顧客との関係性です」→ 経営層「うーん...」→ 結論出ず
  - 本当のコアコンピタンス: 会議を3時間やっても結論を出さずに次回に持ち越せる能力

- **シナジー効果** → 「1+1=2」じゃなく「1+1=3」になる魔法。なおエビデンスはない
  - 合併、提携、新規事業...全てに「シナジー効果が期待できる」と付けると通る
  - エンジニア「システム統合で工数が3倍になります」→ 経営層「でもシナジーで相殺されるから」→ されません
  - シナジーという言葉を使った時点で、誰も定量的な根拠を求めなくなる便利ワード

**ウケる仕掛け:**

- 「ビジネス用語ビンゴカード」— 会議中に5つ揃ったら帰っていい(帰れない)
- 「エンジニアのための会議サバイバルガイド」— とりあえず「検討します」と言え
- 「翻訳例: ビジネス語→日本語→エンジニア語」— 三段階翻訳でようやく分かる
- 「『技術的には可能ですが...』のその後の正しい続け方10選」— 全部断り文句

**コラム:**

- 「なぜビジネス用語はカタカナなのか」— 英語にすると説得力が3割増しになる不思議
- 「会議で決まったこと vs Slackで決まったこと」— 本当の意思決定はどこで?
- 「エンジニアとビジネス職の『緊急』の定義の違い」— お前の緊急は俺の通常運転
- 「技術的負債をROIで説明する方法」— 無理を可能にする詭弁術

### 第4章:「アーキテクチャで雰囲気で使ってる」系

**特徴:** 設計の話で出てくる謎ワード

- **疎結合と密結合** → どっちがいいの?
- **ステートレスとステートフル** → 状態を持つ/持たないの何が違う?
- **イミュータブル** → 不変…だから何?
- **ポリモーフィズム** → 多態性って言われても
- **デプリケート** → 非推奨と廃止の違い
- **ペイロード** → 荷物?データ本体?
- **エンドポイント** → 終点?いやAPIの…
- **スロットリング** → 絞る?制限?

**ウケる仕掛け:** 「設計レビューで恥をかかないための用語チェックリスト」

### 第5章:「インフラ・ネットワークで雰囲気で使ってる」系

**特徴:** SREとかDevOpsとか

- **オーケストレーション** → コンテナの指揮者?
- **プロビジョニング** → 準備?設定?構築?
- **イングレスとエグレス** → 入口と出口、だけ?
- **ロードバランサー** → 負荷分散は分かるけど仕組みは?
- **リバースプロキシ** → 何が逆なの?
- **サイドカー** → バイクの横?コンテナの話?
- **メッシュネットワーク** → 網目状…で?
- **サーキットブレーカー** → ブレーカー落とす?

**ウケる仕掛け:** 「インフラ担当との会話で困らない用語マップ」

### 第6章:「データベースで雰囲気で使ってる」系

**特徴:** SQL書けるけど説明できない

- **トランザクション** → 処理のかたまり…?
- **ACID特性** → 酸性じゃないよ
- **正規化** → 第1、第2、第3…覚えてます?
- **インデックス** → 速くなるやつ、以上
- **シャーディング** → 分割?パーティショニングとの違いは?
- **レプリケーション** → 複製してるのは知ってる
- **オプティミスティックロック** → 楽観的…って何が?
- **N+1問題** → Nって何やねん

**ウケる仕掛け:** 「DBチューニング会議で生き残る最低限の知識」

### 第7章:「セキュリティで雰囲気で使ってる」系

**特徴:** なんとなく怖いやつ

- **サニタイズとバリデーション** → 消毒と検証?
- **CSRF、XSS、SQLインジェクション** → 略語の元は?
- **ゼロトラスト** → 信用しない…誰を?
- **ソルト** → パスワードに塩?
- **ノンス** → 使い捨て番号…なぜ?
- **オリジン** → 起源?ドメインのこと?
- **CORS** → クロスオリジン…何?
- **ホワイトリスト/ブラックリスト** → 今は言っちゃダメ?

**ウケる仕掛け:** 「脆弱性診断レポートを読むための用語集」

### 第8章:「パフォーマンスで雰囲気で使ってる」系

**特徴:** 速い/遅いの話で出てくる謎指標

- **レイテンシとスループット** → 遅延と処理量…の違いは?
- **キャッシュヒット率** → 高い方がいいのは分かる
- **ボトルネック** → 瓶の首?
- **デグレード** → 劣化?デグレ?
- **メモリリーク** → 漏れてる…どこに?
- **GC(ガベージコレクション)** → ゴミ収集、いつ?
- **プロファイリング** → 犯罪捜査じゃないよ
- **ベンチマーク** → 測定してるのは何?

**ウケる仕掛け:** 「遅い!って言われたときの調査フローチャート」

### 第9章:「CI/CDで雰囲気で使ってる」系

**特徴:** パイプライン作ってるけど…

- **アーティファクト** → 成果物?どの段階の?
- **ブルーグリーンデプロイ** → 色の意味は?
- **カナリアリリース** → カナリア?炭鉱?
- **ロールバック** → 戻すのは分かる、どこまで?
- **イミュータブルインフラ** → 不変…壊して作る?
- **Feature Flag** → フラグ?トグル?
- **Smoke Test** → 煙?どういうテスト?
- **ヘルスチェック** → 何の健康診断?

**ウケる仕掛け:** 「デプロイ失敗時に慌てないための用語理解」

### 第10章:「コードレビューで雰囲気で使ってる」系

**特徴:** レビューコメントの謎ワード

- **LGTM** → Looks Good To Me、でも本当に?
- **IMO、IMHO** → 個人的意見…の略?
- **nit** → ニット?細かい指摘?
- **WIP** → Work In Progress、プルリクに書くけど
- **YAGNI** → You Aren’t Gonna Need It、いつ使う?
- **DRY** → Don’t Repeat Yourself、どこまで?
- **SOLID原則** → 頭文字…全部言える?
- **コードスメル** → 臭い?具体的には?

**ウケる仕掛け:** 「レビュアーの真意を読み解く」会話例

### 第11章:「モダン開発で雰囲気で使ってる」系

**特徴:** 最近よく聞くバズワード的な

- **Serverless** → サーバーないの?あるの?
- **Edge Computing** → エッジ…端っこ?
- **Observability** → 可観測性って何観測?
- **Gitops** → Gitで運用…?
- **IaC(Infrastructure as Code)** → コード化の本質は?
- **Service Mesh** → マイクロサービスの網?
- **Jamstack** → ジャムの山?
- **Monorepo vs Multirepo** → どっちがいいの論争

**ウケる仕掛け:** 「技術選定会議で知ったかぶりしないための最新用語」
