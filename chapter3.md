# 第3章:「ビジネス会議で雰囲気で使ってる」系

## はじめに:エンジニア、会議室という戦場へ

「次の経営会議、開発チームから誰か出てくれない?」

この一言を聞いた瞬間、Slackのチャンネルが突然静まり返った経験はないだろうか。誰もが「既読無視」を決め込み、5分後には「あ、今日MTG多くて…」という言い訳が飛び交う。

なぜか。理由は簡単だ。**ビジネス会議は、エンジニアにとって外国語だらけの異世界だからである。**

「ROIを最大化しつつKPIを達成し、ステークホルダーとのコンセンサスを取りながらピボットする」

…日本語のはずなのに、何一つ頭に入ってこない。まるで呪文だ。そして最悪なのは、**周りの人たちは全員わかった顔をしている**ことである。

本章では、そんなエンジニアの皆さんが経営会議という戦場で生き残るために必要な「ビジネス用語」を、皮肉と現実を交えながら解説していく。安心してほしい。周りの人たちも、実は半分くらいは雰囲気で使っているから。

## ROI(Return on Investment):投資対効果の魔法の数字

**雰囲気使用例:**
「この機能追加、ROIどうなの?」
「え…と、効率良くなるので…ROI高いです!(たぶん)」

**正しい理解:**
ROI = (得られた利益 - 投資コスト) ÷ 投資コスト × 100%

### エンジニアが直面する現実

「サーバーのメモリ増設したいんですが」
「ROIは?」

この瞬間、エンジニアの脳内では高速計算が始まる。メモリ代は10万円。これによって得られる利益は…？処理速度が30%向上して、ユーザー満足度が上がって、離脱率が下がって…**そんなの数値化できるかー!!**

しかし、ビジネスサイドにとってROIは**すべての判断基準**である。「技術的に必要です」だけでは通らない。彼らが知りたいのは「それでいくら儲かるの?」という一点のみ。

### エンジニア流・ROIの伝え方

❌ 「メモリ不足でパフォーマンスが悪いので増設したいです」
⭕ 「現在、ピーク時にページ表示が3秒かかり、直帰率が40%。業界標準の1秒に改善すれば直帰率を25%に下げられ、月間で約300万円の機会損失を防げます。投資10万円で年間ROI 3,500%です」

**ポイント:** 数字で語れ。そして年間換算すると大きく見える。

### コラム:ROIで語れないものたち

- セキュリティ対策(被害額は不明)
- 技術的負債の解消(効果が見えづらい)
- リファクタリング(「何も変わってないじゃん」と言われる)

これらは「**リスク回避のROI**」という概念で攻めよう。「やらなかった場合の損失」を語るのだ。

## KPI vs KGI vs OKR:目標設定の三兄弟

**雰囲気使用例:**
「今期のKPIは?」
「え、KGI?OKR?どれのこと?」

**正しい理解:**
- **KGI**(Key Goal Indicator):最終目標。「売上1億円」「ユーザー10万人」など
- **KPI**(Key Performance Indicator):KGI達成のための中間指標。「月間アクティブユーザー2万人」「コンバージョン率3%」など
- **OKR**(Objectives and Key Results):目標と主要な成果。「〇〇を達成する(Objective)ために、△△という成果を出す(Key Results)」

### 三兄弟の関係性

```
KGI:「年商10億円」(最終ゴール)
 ↑
KPI:「月間新規ユーザー5,000人」「月間売上8,000万円」(中間指標)
 ↑
OKR:O「サービス認知度を向上させる」→ KR「SNSフォロワー1万人」「メディア掲載10件」
```

### エンジニアがブチギレるポイント

**「今期のKPIはバグゼロで!」**

待ってくれ。それはKPIではなく、単なる**願望**だ。KPIは「測定可能な指標」であるべきで、「バグゼロ」は非現実的すぎる。

正しくは:「クリティカルバグの解決時間を平均24時間以内にする」「月間バグ発生数を前月比20%削減」など。

### エンジニアあるある:数値化の罠

```
ビジネス:「コードの品質をKPIにしよう」
エンジニア:「コードカバレッジ90%とか?」
ビジネス:「さすが!具体的でいいね!」

3ヶ月後…

エンジニア:「カバレッジ90%達成しました!」
ビジネス:「でもバグ増えてない?」
エンジニア:「……(テストの質は問われてなかったんだが)」
```

**教訓:** 数値は嘘をつかないが、都合の良い数値を選ぶことはできる。

## ステークホルダー:会議に呼ばれる人が増える呪文

**雰囲気使用例:**
「ステークホルダーに確認してから進めよう」
「(誰に確認するんだ…?)」

**正しい理解:**
利害関係者。プロジェクトに影響を与える、または影響を受けるすべての人や組織。

### エンジニアの脳内翻訳

「ステークホルダーに確認」=「誰か偉い人に聞いてから」

しかし問題は、**誰が**ステークホルダーなのかが曖昧なことだ。

- プロダクトオーナー?
- マネージャー?
- 営業部?
- カスタマーサポート?
- エンドユーザー?
- CEO?
- 株主?

**全員だ。**

### ステークホルダーが増えると何が起きるか

```
ステークホルダー1人 → 1日で決定
ステークホルダー3人 → 1週間で決定
ステークホルダー5人 → 1ヶ月で決定
ステークホルダー10人 → 決定しない
```

これを「**ステークホルダーの法則**」と呼ぶ(今考えた)。

### エンジニアの護身術

「〇〇の件、ステークホルダーは誰ですか?」と**最初に明確にする**。

さもないと、実装が終わった後に「あ、法務にも確認必要だったわ」と言われ、全ボツになる地獄が待っている。

## リソース:人か?金か?時間か?サーバーか?

**雰囲気使用例:**
「リソース足りないんだよね〜」
「(何が足りないんだ?)」

**正しい理解:**
文脈によって意味が変わる魔法の言葉。

### リソースの種類

| 文脈 | 意味 |
|------|------|
| 「リソースが足りない」 | 人手が足りない |
| 「リソースを確保してほしい」 | 予算をくれ |
| 「リソースを割けない」 | 時間がない |
| 「リソース使用率が高い」 | CPU・メモリが圧迫されている |
| 「リソースの最適化」 | よくわからないけど効率化したい |

### エンジニアとビジネスの断絶

```
ビジネス:「来月のプロジェクト、リソースある?」
エンジニア:「サーバーリソースは大丈夫です」
ビジネス:「いや、人的リソース」
エンジニア:「……人って言ってくれ」
```

**なぜ「人」と言わずに「リソース」と言うのか?**

それは、人間を「交換可能な資源」として扱う経営的思考の表れである。エンジニアA氏が休んでも、エンジニアB氏を投入すれば同じ成果が出る、という幻想。

現実:「A氏しかこのレガシーコード読めないんですけど」

### コラム:無限リソースの幻想

```
ビジネス:「人増やせば早くなるよね?」
エンジニア:「ブルックスの法則って知ってます?」
ビジネス:「?」
エンジニア:「遅れているプロジェクトに人を追加すると、さらに遅れるんです」
ビジネス:「じゃあどうしろと」
エンジニア:「最初から適切な人数を割り当ててください」
```

## マイルストーン vs スプリント:時間の区切り方

**雰囲気使用例:**
「次のマイルストーンまでに終わらせて」
「スプリントで区切ろう」
「(どっち?)」

**正しい理解:**
- **マイルストーン**: プロジェクト全体の重要な節目。「β版リリース」「正式ローンチ」など
- **スプリント**: 短期間(通常1〜2週間)の開発サイクル(アジャイル用語)

### ウォーターフォール派 vs アジャイル派の戦い

ビジネス会議では、この2つの世界観が混在している。

**ウォーターフォール脳:**
「要件定義→設計→実装→テスト→リリース」という一本道。マイルストーンは「各工程の完了」。

**アジャイル脳:**
「小さく作って→確認して→直して」を繰り返す。スプリントごとに動くものを出す。

問題は、**ビジネスサイドが両方の用語を混ぜて使う**ことだ。

```
ビジネス:「スプリントごとにマイルストーン達成してね」
エンジニア:「それはおかしい」
ビジネス:「なんで?」
エンジニア:「マイルストーンは大きな節目で、スプリントは小刻みなサイクルです。毎スプリントにマイルストーンがあったら、マイルストーンの意味がない」
ビジネス:「じゃあ、スプリントゴールね」
エンジニア:「それなら正しいです」
```

### マイルストーンの罠

「12月末をマイルストーンにしよう!」

**エンジニアの心の声:** 「年末はリリースしたくないんだが…誰がクリスマスにバグ対応するんだ」

**現実的なマイルストーン設定:**
- 年末年始・GW・お盆を避ける
- 金曜リリースを避ける(週末に障害対応したくない)
- 大型連休前を避ける(障害時に人がいない)

しかし、これを言うと「エンジニアは後ろ向き」と言われる理不尽。

## ピボット:方向転換か?仕様変更か?

**雰囲気使用例:**
「ユーザーの反応が悪いから、ピボットしよう」
「(また1から作り直しかよ…)」

**正しい理解:**
事業戦略の大幅な方向転換。単なる機能追加や仕様変更ではなく、**ビジネスモデルそのものを変える**こと。

### 本物のピボット vs 偽物のピボット

**本物のピボット:**
- Instagram:位置情報SNS → 写真共有SNS
- Twitter:ポッドキャスト配信 → 短文SNS
- YouTube:出会い系サイト → 動画共有サイト

**偽物のピボット(ただの仕様変更):**
- 「ボタンの色を青から緑にピボット」← それは仕様変更
- 「ログイン機能をメールからSNS連携にピボット」← それは改善
- 「スマホ対応にピボット」← それは当然の対応

### エンジニアが疲弊する「ピボット乱用」

```
月曜日:「BtoC向けサービスで行こう!」
水曜日:「やっぱりBtoBにピボットしよう」
金曜日:「BtoBtoC(何それ)にピボットだ!」
翌週月曜:「最初のBtoCに戻そう(それピボットバックって言わない?)」
```

**エンジニアの限界:**
「ピボットは構わん。だが1ヶ月に3回もピボットするな。それは優柔不断というんだ」

### ピボットの適切な使い方

本来、ピボットは**データに基づいた戦略的判断**である。

- ユーザーインタビューで方向性の誤りが判明
- 市場調査で別のニーズを発見
- 競合の出現で差別化が必要に

感情や思いつきで「ピボット」と言うのは、ただの**朝令暮改**である。

## MVP(Minimum Viable Product):どこまで削るか問題

**雰囲気使用例:**
「まずはMVPで」
「(ログイン機能もないけど、これMVPって言えるのか?)」

**正しい理解:**
実用最小限の製品。ユーザーに価値を提供しつつ、最小限の機能で**仮説検証**するためのもの。

### MVPの本質を勘違いした結果

❌ **間違ったMVP観:**
「とにかく早く作れば良い」→ バグだらけの半完成品
「機能を削りまくる」→ 誰も使えないゴミ

⭕ **正しいMVP観:**
「仮説を検証できる最小限の機能」→ 小さいが動く、価値を提供する

### MVPの有名な図解

```
❌ 悪いMVP:
タイヤ → シャーシ → 車体 → やっと車
(最後まで乗れない)

⭕ 良いMVP:
スケボー → キックボード → 自転車 → バイク → 車
(各段階で「移動できる」という価値を提供)
```

### エンジニアとビジネスの認識ズレ

```
ビジネス:「MVP作って!」
エンジニア:「わかりました」

(2週間後)

エンジニア:「MVPできました」
ビジネス:「え、デザインこれだけ?」
エンジニア:「MVPなので」
ビジネス:「ログイン機能ないの?」
エンジニア:「MVPなので」
ビジネス:「これじゃ使えないよ!」
エンジニア:「MVPなので」
```

**問題の本質:** MVPの定義が曖昧

### MVPの定義を最初にすり合わせる

- **目的:** 何の仮説を検証するのか?
- **ターゲット:** 誰に使ってもらうのか?
- **成功指標:** どうなったら次のステップに進むのか?
- **期間:** いつまでに作るのか?
- **削る機能:** 何を諦めるのか?

これを決めずにMVPを作ると、「思ってたのと違う」が100%発生する。

## コンセンサス:全員一致?それとも多数決?

**雰囲気使用例:**
「コンセンサス取れた?」
「(誰の承認が必要なんだ?)」

**正しい理解:**
合意形成。ただし「全員が賛成」ではなく、「全員が納得して受け入れた状態」。

### 日本企業の会議あるある

```
1時間の会議
├─ 50分:雑談と状況共有
└─ 10分:「じゃあ、これで良いですかね?」「異議なし」

これをもって「コンセンサスが取れた」とする。
```

**しかし…**

会議後のSlack:「さっきの件、やっぱりおかしいと思うんですよね」

**それ、会議で言えよ!!**

### コンセンサスの種類

1. **全員一致:** 全員が賛成(理想だが非現実的)
2. **多数決:** 過半数が賛成(民主的だが少数派が不満)
3. **消極的同意:** 積極的反対はない(日本でよく見る)
4. **独裁:** 偉い人が決める(一番早い)

### エンジニアが求める「真のコンセンサス」

```
❌ 形だけのコンセンサス:
「反対ある人?」「…(シーン)」「じゃあ決定で」

⭕ 実質的なコンセンサス:
「懸念点は?」→ 各自意見 → 議論 → 調整 → 全員が納得
```

**時間はかかるが、後でひっくり返らない。**

### コンセンサスが取れない時の対処法

**Option 1:** 期限を決める
「〇日までに反対意見がなければ進める」

**Option 2:** 意思決定者を明確にする
「最終的には〇〇さんが決める」

**Option 3:** 実験的に試す
「まず1ヶ月やってみて、ダメなら戻す」

## グロースハック:成長のための小技集

**雰囲気使用例:**
「グロースハックで伸ばそう!」
「(具体的に何するの?)」

**正しい理解:**
製品の成長を促進するための創意工夫。データ分析とマーケティングと開発を組み合わせた手法。

### グロースハックの有名事例

**Dropbox:**
「友達を紹介したら、両方に容量プレゼント」→ 爆発的にユーザー増

**Hotmail:**
メールの最後に「PS: I love you. Get your free email at Hotmail」→ メール自体が広告に

**Airbnb:**
Craigslistに自動投稿する機能を追加 → 既存ユーザーベースを活用

### 日本企業の「自称グロースハック」

```
マネージャー:「グロースハックしよう!」
エンジニア:「具体的には?」
マネージャー:「とにかく数字を伸ばす!」
エンジニア:「それはグロースハックじゃなくて、ただの目標では」
```

### グロースハックと単なる機能追加の違い

**グロースハック:**
- データに基づく仮説
- 低コストで実装可能
- 成長に直結する施策
- 測定可能な効果

**単なる機能追加:**
- 「あったら便利かも」レベル
- 工数がかかる
- 効果が不明
- 誰も使わない

### エンジニアが知っておくべきグロースハックの基本

**AARRRモデル(海賊指標):**
1. **Acquisition**(獲得):どうやってユーザーを連れてくるか
2. **Activation**(活性化):初回体験を良くする
3. **Retention**(継続):また使ってもらう
4. **Referral**(紹介):友達に教えてもらう
5. **Revenue**(収益):お金を払ってもらう

「グロースハック」と言われたら、このどこを改善するのか聞こう。

## KPI地獄からの脱出:コラム

### KPI設定で起きる悲劇

```
経営:「ユーザー数を増やそう!」
→ KPI:「月間新規登録ユーザー1万人」

3ヶ月後…

結果:登録ユーザー1万人達成!
しかし:アクティブユーザー100人

経営:「なんで誰も使ってないの?」
現場:「登録数がKPIだったので…」
```

**教訓:** KPIは、意図しない行動を誘発する。

### 悪いKPIの例

| KPI | 起きる悲劇 |
|-----|----------|
| コード行数 | 無駄に長いコードが量産される |
| バグ解決数 | 簡単なバグばかり直して重要なバグを放置 |
| 機能リリース数 | 誰も使わない機能を乱発 |
| 会議時間削減 | 必要な議論まで削られる |

### 良いKPIの条件

1. **測定可能:** 数値で表せる
2. **影響可能:** 自分たちの行動で変えられる
3. **ビジネス価値と連動:** 数字が良くなる=ビジネスが良くなる
4. **バランスが取れている:** 1つの指標に偏らない

## 技術的には可能ですが…:翻訳ガイド

エンジニアが会議で使う**魔法の枕詞**、それが「技術的には可能ですが…」

この「…」の後に続く言葉で、すべてが決まる。

### パターン別・翻訳表

| エンジニアの発言 | 本当の意味 |
|----------------|-----------|
| 技術的には可能ですが、工数がかかります | できるけど、やりたくない |
| 技術的には可能ですが、リスクがあります | できるけど、失敗したら責任取ってくれるの? |
| 技術的には可能ですが、他の優先度が… | できるけど、もっと大事なことあるよね? |
| 技術的には可能ですが、保守が大変です | できるけど、後で地獄を見るよ |
| 技術的には可能ですが、仕様が不明確です | 何を作ればいいか、ちゃんと説明してくれ |
| 技術的には可能ですが、パフォーマンスが… | できるけど、クソ遅くなるよ |
| 技術的には可能ですが、セキュリティ的に… | できるけど、炎上するよ |

### ビジネスサイドの翻訳

| ビジネスの発言 | 本当の意味 |
|--------------|-----------|
| ささっと作って | 簡単だと思ってる |
| ちょっとした修正 | たぶん大工事 |
| 他社もやってるよね | 調べてない |
| ユーザーが求めてる | 自分が欲しい |
| 今日中にできる? | 明日プレゼンなの |
| これくらいできるよね? | できないとは言わせない |
| 柔軟に対応して | 仕様は決めてない |

### 建設的な会話のために

❌ **悪い例:**
```
ビジネス:「この機能、今日中にできる?」
エンジニア:「無理です」
ビジネス:「なんで?」
エンジニア:「無理だから」
```

⭕ **良い例:**
```
ビジネス:「この機能、今日中にできる?」
エンジニア:「フル実装は5日かかりますが、コアな部分だけなら明日までに動くものを作れます。何を優先しますか?」
ビジネス:「コアな部分って?」
エンジニア:「〇〇と△△です。××は後回しにできます」
```

**ポイント:** 代替案を出す。二者択一ではなく、グラデーションで考える。

---

## まとめ:エンジニアよ、ビジネス用語を恐れるな

ビジネス用語は、確かに抽象的で曖昧で、時に意味不明だ。

しかし、**彼らも必死なのだ。**

エンジニアが「レイテンシ」「スループット」「スケーラビリティ」を当たり前に使うように、ビジネスサイドも「ROI」「KPI」「ステークホルダー」を当たり前に使う。

大事なのは、**お互いの言語を理解しようとする姿勢**である。

### 会議で生き残るための3箇条

1. **曖昧な用語は、その場で定義を確認する**
   「〇〇って、具体的には?」と聞くのは恥ずかしいことじゃない

2. **技術的な話をビジネス価値に翻訳する**
   「パフォーマンス改善」→「ユーザー体験向上→離脱率低下→売上増」

3. **No と言うときは代替案をセットで**
   「できません」ではなく「Aは難しいですが、Bならできます」

### 最後に

「次の経営会議、誰か出てくれない?」

今度この質問が来たら、少しだけ手を挙げやすくなっているはずだ。

**ビジネス用語は呪文じゃない。ただの道具だ。**

道具の使い方を知れば、エンジニアは会議という戦場で、ちゃんと戦える。
