# 第3章:「ビジネス会議で雰囲気で使ってる」系

## はじめに

ROI、KPI、ステークホルダー...ビジネス会議で飛び交うこれらの用語。エンジニアとして技術は得意でも、経営会議やプロダクトミーティングで出てくるビジネス用語に「え、それって何?」と内心焦ったことはありませんか？この章では、エンジニアが最も苦手とする「ビジネス会議で雰囲気で使ってる」用語を徹底解説します。技術とビジネスの橋渡しができるエンジニアになりましょう!

---

## ROI - 投資対効果…技術選定でどう使う?

### 雰囲気使用例

```
「この新技術導入、ROI良さそうですね!」
（なんとなく「良い」と言いたいだけ）
```

### 正しい使用例

```
「クラウド移行の初期投資は500万円ですが、年間運用コストが
300万円削減できるので、2年でROIが100%を超えます」
```

### 何が違う?

ROI（Return On Investment）は:
**「投資したお金に対して、どれだけリターン（利益）があるか」を示す数値**

計算式:
```
ROI = (得られた利益 - 投資コスト) ÷ 投資コスト × 100%
```

例:
- 投資: 100万円のツール導入
- 効果: 年間50万円の工数削減
- ROI = (50 - 100) ÷ 100 = -50%（1年目）
- ROI = (100 - 100) ÷ 100 = 0%（2年目で回収）
- ROI = (150 - 100) ÷ 100 = 50%（3年目）

### エンジニア向けの使い方

技術選定でROIを考える:
- **CI/CDツール導入**: 初期コストvs手動デプロイの工数削減
- **モニタリング強化**: ツール費用vs障害対応時間の短縮
- **リファクタリング**: 作業時間vs将来の開発速度向上

### 現場での注意点

- 「ROI良い」だけでは不十分。具体的な数字で説明する
- 技術投資のROIは「工数削減」「障害減少」「開発速度」などで測る
- 短期的なROIだけでなく、長期的な視点も重要

## KPI、KGI、OKR - 全部目標っぽいけど違いは?

### 雰囲気使用例

```
「今月のKPI、達成しましたね!」
（売上目標のことを言ってる）
```

### 正しい使用例

```
「KGI（最終目標）は年間売上10億円、そのためのKPI（中間指標）として
月間アクティブユーザー10万人を設定しています」
```

### 何が違う?

これらは全て「目標管理」の用語ですが、階層が違います:

**KGI (Key Goal Indicator) = ゴール（最終目標）**
- 「何を達成したいか」
- 例: 売上10億円、シェア20%

**KPI (Key Performance Indicator) = プロセス指標**
- 「KGI達成のための中間指標」
- 例: 月間ユーザー数、コンバージョン率、バグ発生率

**OKR (Objectives and Key Results) = 目標と成果指標**
- O（目標）: 定性的な挑戦的目標
- KR（成果指標）: Oを測る3〜5個の定量指標
- 例: O「最高のユーザー体験を提供」→ KR1「NPS 50以上」KR2「離脱率5%以下」

### 関係性の図

```
KGI（最終ゴール）
  ↑
 KPI（中間指標） ← これを測って進捗管理
  ↑
日々の活動

OKRは別の枠組み:
Objective（野心的目標）
  ↑
Key Results（測定可能な成果）
```

### エンジニア的な例

**KGI**: サービス可用性99.9%
**KPI**:
- インシデント発生件数（月5件以下）
- 平均復旧時間（30分以内）
- デプロイ頻度（週3回以上）

**OKR例**:
- O: 世界最速のAPIを提供する
- KR1: レスポンスタイム平均100ms以下
- KR2: P99レイテンシ500ms以下
- KR3: エラーレート0.1%以下

### 現場での注意点

- KPIは「手段」、KGIは「目的」。手段の最適化に偏らない
- OKRは「60〜70%達成で成功」とする文化（挑戦的目標だから）
- KPIを増やしすぎると管理不能に（3〜5個程度が目安）

## ステークホルダー - 利害関係者…誰が該当?

### 雰囲気使用例

```
「ステークホルダーに確認取ります」
（上司のことを言ってる）
```

### 正しい使用例

```
「この機能変更のステークホルダーは、ユーザー、営業チーム、
カスタマーサポート、経営陣です。それぞれに影響を説明します」
```

### 何が違う?

ステークホルダー（Stakeholder）は:
**「プロジェクトや意思決定に利害関係を持つ全ての人・組織」**

"Stake"は「杭を打つ」→「関与する」という意味。

### プロジェクトのステークホルダー例

**内部:**
- 開発チーム
- プロダクトマネージャー
- デザイナー
- マーケティング
- カスタマーサポート
- 経営層

**外部:**
- エンドユーザー
- クライアント企業
- パートナー企業
- 投資家

### エンジニアが見落としがちなステークホルダー

- **カスタマーサポート**: 機能変更で問い合わせが増える
- **法務・コンプライアンス**: データ保管やプライバシー関連
- **運用チーム**: デプロイや監視への影響
- **他チームの開発者**: API変更の影響を受ける

### 現場での注意点

- 「誰に影響があるか」を最初に洗い出す
- 技術的な変更でも、ビジネス側のステークホルダーに影響することがある
- ステークホルダーの優先順位をつける（全員の要望は聞けない）

## リソース - 人?お金?時間?サーバー?

### 雰囲気使用例

```
「リソースが足りないので来月まで待ってください」
（人手不足のことを言ってる）
```

### 正しい使用例

```
「このプロジェクトに必要なリソースは、エンジニア2名（人的リソース）、
クラウド費用月30万円（資金リソース）、開発期間3ヶ月（時間リソース）です」
```

### 何が違う?

リソース（Resource）は「資源」という意味で、文脈によって指すものが変わります:

**ビジネス文脈:**
1. **人的リソース**: エンジニア、デザイナーなどの人員
2. **資金リソース**: 予算、お金
3. **時間リソース**: 納期、スケジュール
4. **物的リソース**: オフィス、機材

**技術文脈:**
1. **計算リソース**: CPU、メモリ
2. **ストレージリソース**: ディスク容量
3. **ネットワークリソース**: 帯域幅
4. **APIリソース**: RESTful APIのエンドポイント

### 現場での混乱例

```
PM: 「リソース足りてる?」
  → 人員のことを聞いている

エンジニア: 「サーバーリソースは十分です」
  → CPU/メモリのことを答えている

→ 会話が噛み合わない!
```

### 現場での注意点

- 「リソース」だけでは曖昧。「人的リソース」「サーバーリソース」と明示する
- ビジネス会議では基本的に「人・金・時間」のいずれかを指す
- 「リソースが足りない」と言うときは、何のリソースか具体的に説明

## マイルストーン - 石の塚?中間目標?

### 雰囲気使用例

```
「次のマイルストーンは来週です」
（単なる締め切りのことを言ってる）
```

### 正しい使用例

```
「プロジェクトのマイルストーンは、M1: 要件定義完了(4月末)、
M2: プロトタイプ完成(6月末)、M3: β版リリース(8月末)です」
```

### 何が違う?

マイルストーン（Milestone）は:
**「プロジェクトの重要な節目・通過点」**

単なる締め切りとの違い:
- ❌ 「タスク完了の期限」→ デッドライン
- ✅ 「プロジェクト全体の中で意味のある地点」→ マイルストーン

### 語源と背景

"Mile Stone"は道路脇に置かれた「1マイルごとの標石」が語源。旅人が「ここまで来た」という進捗を確認できる目印です。

### マイルストーンの例

**開発プロジェクト:**
1. M1: 技術検証完了（PoC成功）
2. M2: MVP（最小機能版）リリース
3. M3: β版リリース
4. M4: 正式リリース
5. M5: 月間ユーザー1万人達成

**特徴:**
- 各マイルストーンで「成果物」がある
- 「ゴーノーゴー判断」（続けるか中止するか）の決定点
- 外部発表やデモのタイミングになることも

### デッドラインとの使い分け

- **デッドライン**: 「このタスクは金曜までに」（日常的な締め切り）
- **マイルストーン**: 「プロトタイプは6月までに完成させる」（プロジェクトの節目）

### 現場での注意点

- マイルストーンは「プロジェクトの骨格」を示す
- 各マイルストーンで何を達成すべきかを明確にする
- マイルストーンごとにステークホルダーへ報告

## MVP (Minimum Viable Product) - 最小限…どこまで削る?

### 雰囲気使用例

```
「とりあえずMVPで出しましょう!」
（未完成品を出す言い訳にしてる）
```

### 正しい使用例

```
「MVPでは、ユーザー登録とコア機能の投稿だけを実装します。
これでユーザーの反応を見て、次の機能開発の優先度を決めます」
```

### 何が違う?

MVP（Minimum Viable Product）は:
**「最小限の機能で、ユーザーに価値を提供し、学びを得られるプロダクト」**

重要なのは3つの要素:
1. **Minimum（最小限）**: 必要最低限の機能
2. **Viable（実用可能）**: ちゃんと動いて価値がある
3. **Product（製品）**: リリースできる品質

### よくある誤解

❌ **MVPは低品質でOK**
- 「Viable」は「実用可能」という意味。バグだらけはNG

❌ **MVPは機能が少ないだけ**
- 目的は「学習」。ユーザーフィードバックを得ることが重要

✅ **MVPは仮説検証のツール**
- 「このコア機能に価値があるか?」を確かめる

### MVP設計の例

**SNSアプリのMVP:**

含める:
- ユーザー登録/ログイン
- 投稿機能（テキストのみ）
- タイムライン表示

含めない:
- 画像投稿（後で追加）
- いいね機能（後で追加）
- 通知機能（後で追加）
- プロフィールカスタマイズ（後で追加）

**判断基準**: 「投稿して見れる」というコア価値を最速で検証

### PMFとの関係

MVP → ユーザー反応を見る → 改善 → **PMF（Product Market Fit）達成**

MVPは「PMFを見つけるための第一歩」

### 現場での注意点

- 「MVPだから適当でいい」は間違い。品質は保つ
- MVPは「削る技術」。何を削るかが重要
- ビジネス側に「MVPは学習ツール」と理解してもらう

## PMF (Product Market Fit) - 市場適合…エンジニアに関係ある?

### 雰囲気使用例

```
「PMF達成したらしいですよ」
（売上が伸びてるらしい、くらいの理解）
```

### 正しい使用例

```
「ユーザーの40%が『このサービスがなくなったら非常に困る』と
回答しているので、PMFに到達したと判断できます」
```

### 何が違う?

PMF（Product Market Fit）は:
**「作った製品が、市場（顧客）のニーズに完全にマッチした状態」**

Marc Andreessen（Netscape創業者）の定義:
> "Product/market fit means being in a good market with a product that can satisfy that market"

### PMFの兆候

定性的:
- ユーザーが自発的に広めてくれる（口コミが発生）
- 解約率が極端に低い
- 「なくなったら困る」という声が多い

定量的:
- **40%ルール**: 「このサービスがなくなったら非常に困る」が40%以上
- リテンション率（継続率）が高い
- オーガニック成長（広告なしで成長）

### なぜエンジニアに関係あるのか?

**PMF前**:
- 仮説検証が中心
- 機能の追加・削除が頻繁
- スピード重視
- 技術負債は許容される

**PMF後**:
- スケール（拡大）が中心
- 既存機能の改善
- 品質・パフォーマンス重視
- 技術負債の返済フェーズ

→ エンジニアリングの優先順位が変わる!

### 現場での注意点

- PMF前に過度な最適化は不要（どうせピボットする）
- PMF達成の判断を誤ると、スケールで失敗する
- エンジニアも「なぜ今これを作るのか」をPMFの文脈で理解すべき

## バーンダウンチャート - 燃え尽き…グラフ?

### 雰囲気使用例

```
「バーンダウン見せてください」
（進捗報告のグラフ、くらいの認識）
```

### 正しい使用例

```
「バーンダウンチャートを見ると、残作業が理想線より上にあるので、
スコープ調整が必要そうです」
```

### 何が違う?

バーンダウンチャート（Burndown Chart）は:
**「残作業量を時系列で示したグラフ」**

"Burn down"は「燃やし尽くす」→「残作業を減らしていく」イメージ。

### グラフの見方

```
残
作 |＼
業 | ＼ 理想線（均等に減る）
量 |  ＼
   |   ●＼    ● 実績線
   |      ●＼
   |        ●
   +----------→ 時間
   開始      終了
```

**読み取れること:**
- 実績が理想線より上 → 遅れている
- 実績が理想線より下 → 順調（or 見積もりが甘かった）
- 実績が横ばい → 作業が進んでいない

### スクラムでの使い方

- **スプリント バーンダウン**: そのスプリント内の残作業
- **リリース バーンダウン**: リリースまでの残作業

毎日更新して、デイリースタンドアップで確認。

### 対比: バーンアップチャート

バーンダウンの逆で「完了した作業」を積み上げるグラフ。
- モチベーション向上に良い（成果が見える）
- スコープ変更が可視化しやすい

### 現場での注意点

- バーンダウンは「早期警告システム」。遅れに早く気づくためのツール
- グラフが理想線と違うのは普通。重要なのは「なぜズレたか」の分析
- 作業を細かく分割しないと、グラフが階段状になって意味がない

## エスカレーション - 上申?どのタイミングで?

### 雰囲気使用例

```
「これ、エスカレしといて」
（上司に報告する、くらいの意味で使ってる）
```

### 正しい使用例

```
「このバグは自チームで解決できないため、インフラチームに
エスカレーションします。判断と対応をお願いします」
```

### 何が違う?

エスカレーション（Escalation）は:
**「自分の権限や能力では解決できない問題を、上位者・専門家に引き継ぐこと」**

"Escalate"は「エスカレーターで上がる」→「上位に上げる」という意味。

### エスカレーションの種類

**階層的エスカレーション:**
- 自分 → 上司 → 部長 → 役員
- 権限や予算が必要な意思決定

**機能的エスカレーション:**
- 開発チーム → インフラチーム
- 営業 → カスタマーサポート
- 専門知識が必要な問題

### エスカレーションすべきタイミング

✅ すべき:
- 自分の権限で判断できない（予算超過、仕様変更など）
- 専門外の知識が必要（セキュリティ、法務など）
- 締め切りに間に合わない見込み
- 顧客影響が大きいインシデント

❌ すべきでない:
- 自分で調べれば解決できる
- 単なる情報共有（報告で十分）

### 正しいエスカレーション方法

1. **状況説明**: 何が起きているか
2. **影響範囲**: 誰にどんな影響があるか
3. **試したこと**: 自分で何をしたか
4. **求めること**: 判断?リソース?知識?

### 現場での注意点

- エスカレーション = 「丸投げ」ではない。情報整理が重要
- 早すぎるエスカレーションも、遅すぎるエスカレーションも問題
- 「エスカレーションルート」をチームで決めておく

## コンセンサス - 合意形成…全員一致?

### 雰囲気使用例

```
「みんなのコンセンサス取れたから進めます!」
（なんとなく反対されなかったから、くらいの意味）
```

### 正しい使用例

```
「技術選定について、全員が『最善ではないが受け入れ可能』という
コンセンサスに達したので、これで進めます」
```

### 何が違う?

コンセンサス（Consensus）は:
**「全員が同意または納得している状態」**

ただし:
- ≠ 全員が「最高!」と思っている
- ≠ 多数決
- = 「この方向性なら受け入れられる」という合意

### コンセンサスのレベル

1. **完全同意**: 「これがベスト!」（理想だが稀）
2. **支持**: 「良い案だと思う」
3. **留保付き同意**: 「懸念はあるが試す価値はある」
4. **棄権**: 「どちらでも良い」
5. **反対**: 「これは受け入れられない」（コンセンサス不成立）

コンセンサスは2〜4のいずれかを全員が示した状態。

### 多数決との違い

**多数決:**
- 51%が賛成なら決定
- 49%は納得していない
- 実行時に抵抗勢力が生まれる

**コンセンサス:**
- 全員が「やってみよう」と思っている
- 実行時の協力が得られる
- 時間はかかるが、実行速度は速い

### 技術的な意思決定でのコンセンサス

**例: フレームワーク選定**

❌ 間違ったコンセンサス取り:
```
リーダー: 「React使います。異論ある人?」
全員: 「...（沈黙）」
リーダー: 「じゃあ決定!」
```

✅ 正しいコンセンサス取り:
```
リーダー: 「React vs Vue を比較しました（資料共有）。
       それぞれの懸念を聞かせてください」
メンバーA: 「学習コストが気になります」
リーダー: 「研修を2週間設けます。これで進められますか?」
メンバーA: 「それなら大丈夫です」
メンバーB: 「パフォーマンスは問題ないですか?」
リーダー: 「ベンチマーク結果はこちらです（共有）」
メンバーB: 「了解しました」
リーダー: 「では全員、この方向性で進めることに合意ということで」
```

### 現場での注意点

- 沈黙 ≠ 同意。明示的に確認する
- コンセンサスには時間がかかる。重要な決定に絞る
- 「完璧な案」を求めると決まらない。「受け入れ可能」を目指す

## コラム1: エンジニアとビジネス職の「優先度」の認識ズレ問題

### よくあるズレのパターン

**ビジネス側:**
「この機能、最優先でお願い!明日までに!」

**エンジニア側:**
「技術負債の返済が最優先。新機能は後回し」

### なぜズレるのか?

それぞれが見ている「優先度の軸」が違うから:

**ビジネス側の優先度:**
1. 売上インパクト
2. 顧客からの要望の強さ
3. 競合との差別化
4. 締め切り（契約、イベントなど）

**エンジニア側の優先度:**
1. システムの安定性
2. 保守性・拡張性
3. 技術負債の影響
4. 実装難易度

### 翻訳のコツ

**ビジネス → エンジニア:**

```
「これは最優先!」
→ 「売上への影響は?」「いつまでに必要?」を確認
→ 優先度を数値化（ROI、リスクなど）
```

**エンジニア → ビジネス:**

```
「技術負債の返済が必要です」
→ 「このまま放置すると、新機能開発が今の2倍時間かかるようになります」
→ ビジネスインパクトで説明
```

### 実践例

❌ 伝わらない説明:
```
エンジニア: 「DBのN+1問題を解決したいです」
ビジネス: 「?（何の話?）」
```

✅ 伝わる説明:
```
エンジニア: 「今のまま放置すると、ユーザーが増えたときに
        画面の読み込みが3秒→10秒に遅くなります。
        その結果、ユーザーの離脱率が30%上がると予測されます。
        今のうちに2日かけて改修すれば防げます」
ビジネス: 「それは今すぐやろう!」
```

### まとめ

- お互いの「優先度の軸」を理解する
- 技術的な話を「ビジネスインパクト」に翻訳する
- 「なぜ今これをやるのか」を常に説明できるようにする

## コラム2: 技術負債をビジネス用語で説明する方法

技術負債（テクニカルデット）は第2章で説明しましたが、これをビジネス側に理解してもらうのは至難の業。ここでは実践的な説明方法を紹介します。

### パターン1: 金融の借金に例える

```
「今回の実装は『技術的な借金』です。

【元本】
- 本来10日かかる実装を、5日で済ませます
- 5日分の時間を『借りる』

【利息】
- この機能を変更するたび、通常の1.5倍時間がかかります
- 年間で20日分の余計な工数が発生（利息）

【返済】
- 3ヶ月後に5日かけてリファクタリング（元本返済）
- 以降、利息は発生しなくなります

→ 1年以上使う機能なら、今返済した方が得です」
```

### パターン2: 家の修繕に例える

```
「家の雨漏り修理を先送りにするようなものです。

- 今5万円かけて修理すれば終わり
- 放置すると:
  - 1年後: 柱が腐る → 修理費50万円
  - 3年後: 家全体が傾く → 建て替え500万円

技術負債も同じで、放置するほど修正コストが膨らみます」
```

### パターン3: 具体的な数字で示す

```
現状分析:
「現在の技術負債により、新機能開発のスピードが30%低下しています」

このまま放置した場合:
「半年後には50%低下、競合に3ヶ月遅れます」

返済した場合:
「2週間の返済で、開発速度が元に戻ります。
 以降は月に1機能多く出せるようになります」

ROI:
「2週間の投資で、年間24週間分の工数が節約できます（ROI 1100%）」
```

### 避けるべき説明

❌ 専門用語だらけ:
```
「モジュール結合度が高く、循環依存が発生しており、
 凝集度が低いのでリファクタリングが必要です」
```

❌ 曖昧:
```
「コードが汚いので、そのうち直したいです」
```

❌ 脅し:
```
「今すぐ直さないと、システムが崩壊します!」
```

### まとめ

技術負債の説明で重要なのは:
1. **現状**: 今どんな問題があるか
2. **放置した場合の影響**: ビジネスにどう影響するか（売上、スピード、リスク）
3. **解決コスト**: 時間、お金、リソース
4. **解決後のメリット**: ROIで示す

「技術的に正しい」だけでなく、「ビジネス的に合理的」と説明できれば、理解が得られます。

## 章末チェック: あなたの理解度は?

次の文章の誤りを指摘できますか?

1. 「この施策、ROI良さそうだから進めましょう!」
   - ヒント: ROIは定性的?定量的?

2. 「KPIは売上10億円です!」
   - ヒント: KPIとKGIの違いは?

3. 「ステークホルダーは上司と顧客です」
   - ヒント: 他に影響を受ける人は?

4. 「リソースが足りません」
   - ヒント: 何のリソース?

5. 「とりあえずMVPで低品質でも出しましょう!」
   - ヒント: MVPの"Viable"の意味は?

6. 「売上が伸びてるからPMF達成ですね!」
   - ヒント: PMFの判断基準は?

7. 「全員が賛成しなかったけど、コンセンサスは取れた」
   - ヒント: コンセンサスの意味は?

### 解答

1. **誤り**: ROIは具体的な数値で示すべきです。「投資100万円に対し、年間50万円のコスト削減、2年でROI 0%」のように計算する。

2. **誤り**: 「売上10億円」は最終目標なのでKGI。KPIはその途中指標（月間ユーザー数、コンバージョン率など）。

3. **誤り**: カスタマーサポート、運用チーム、法務、マーケティングなど、影響を受ける人は多岐にわたります。洗い出しが重要。

4. **誤り**: 「リソース」だけでは曖昧。人的リソース、資金、時間、サーバーリソースなど、具体的に何が足りないか明示すべき。

5. **誤り**: MVPの"Viable"は「実用可能」という意味。最小限でも品質は保つ必要がある。低品質は許されません。

6. **誤り**: PMFは売上ではなく「ユーザーがなくなったら困る」という状態。40%ルールやリテンション率などで判断します。

7. **誤り（文脈による）**: コンセンサスは「全員が同意または受け入れ可能」という状態。完全に賛成でなくても、「受け入れ可能」ならコンセンサスは成立します。ただし、強い反対があればコンセンサスは不成立。

## 第3章のまとめ

ビジネス用語は、エンジニアにとって「異世界の言語」に感じるかもしれません。しかし、これらの用語を理解することで、経営会議やプロダクトミーティングで「置いてけぼり」にならず、技術的な判断をビジネス価値に翻訳できるようになります。

**キーポイント:**

- **ROI**: 投資対効果を数値で示す。技術投資も工数削減などで計算可能
- **KPI/KGI/OKR**: 目標の階層を理解する。KGI（最終目標）、KPI（途中指標）、OKR（挑戦的目標）
- **ステークホルダー**: 利害関係者を広く捉える。技術変更でもビジネス側に影響あり
- **リソース**: 文脈で意味が変わる。人・金・時間かサーバーリソースか明示する
- **マイルストーン**: プロジェクトの重要な節目。単なる締め切りではない
- **MVP**: 最小限だが実用可能な製品。学習ツールとして使う
- **PMF**: 製品と市場の適合。「なくなったら困る」が40%以上
- **バーンダウンチャート**: 残作業を可視化。早期警告システム
- **エスカレーション**: 上位に上げる。丸投げではなく情報整理が重要
- **コンセンサス**: 全員の合意。沈黙≠同意、明示的確認が必要

次の第4章では、「アーキテクチャで雰囲気で使ってる」系の用語を深掘りします!
