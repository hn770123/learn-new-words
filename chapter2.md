# 第2章:「開発フローで雰囲気で使ってる」系

## はじめに

スクラム、アジャイル、スプリント...開発現場で毎日のように飛び交うこれらの言葉。「わかってます」と言いつつ、実は正確な定義を説明できないまま使っていませんか？この章では、開発フローやプロジェクト管理で「雰囲気で使ってる」用語を徹底解説します。

---

## スプリント - なぜ短距離走?

### 雰囲気使用例

```
「今スプリント、めっちゃ忙しいわ〜」
（単に「今週」とか「今月」の意味で使ってる）
```

### 正しい使用例

```
「今回の2週間スプリントでは、ユーザー認証機能の実装を
完了させることを目標としています」
```

### 何が違う?

スプリントは単なる「期間」ではありません。**「決まった期間内で、決めた機能を完成させる」という約束の単位**です。

- スプリント期間は固定（通常1〜4週間）
- スプリント開始時に「何を作るか」を決める
- スプリント中は原則、目標を変えない
- スプリント終了時には「動くソフトウェア」を必ず作る

### 語源と背景

英語の"Sprint"は「短距離走」の意味。マラソンのように長期間だらだら走るのではなく、短距離を全力で走り切る、というイメージから来ています。アジャイル開発の考え方で「短い周期で確実に成果を出す」ことを重視するため、この名前がつきました。

### 現場での注意点

- 「スプリント = 開発期間」と理解するのは半分正解
- 重要なのは「固定期間」と「完成させる約束」の2つ
- スプリント中に仕様をコロコロ変えるのはアンチパターン

---

## バックログ - 溜まってる仕事…だけ?

### 雰囲気使用例

```
「バックログ溜まってるから、今日は残業だな〜」
（単なるTODOリストと思ってる）
```

### 正しい使用例

```
「プロダクトバックログの上位3項目を、
次のスプリントプランニングで議論しましょう」
```

### 何が違う?

バックログは「やることリスト」ではあるけど、**優先順位がついていて、常に見直される「生きたリスト」**です。

- **プロダクトバックログ**: プロダクト全体でやりたいこと全て
- **スプリントバックログ**: 今回のスプリントでやると決めたこと

重要なのは:

- 優先度が高い順に並んでいる
- 定期的に見直して並び替える
- 上位のアイテムほど詳細に定義されている

### 語源と背景

"Backlog"は「積み残し」という意味。元々は受注残や未処理の注文を指すビジネス用語でした。スクラムでは「まだ実装していない機能要望のリスト」という意味で使われます。

### 現場での注意点

- バックログは「やるべきこと」ではなく「やりたいこと候補」
- 下の方のアイテムは永遠にやらないこともある（それでOK）
- 「バックログを消化する」という表現は微妙（優先度見直しが前提だから）

---

## リファクタリング - リライトとの違いは?

### 雰囲気使用例

```
「この機能、リファクタリングして作り直しますね」
（全部書き直すつもりで言ってる）
```

### 正しい使用例

```
「この関数、リファクタリングして可読性を上げます。
外部から見た動作は一切変わりません」
```

### 何が違う?

リファクタリングの定義は明確です:
**「外部から見た振る舞いを変えずに、内部構造を改善すること」**

つまり:

- ✅ 変数名を分かりやすくする → リファクタリング
- ✅ 重複コードを関数にまとめる → リファクタリング
- ✅ 長い関数を分割する → リファクタリング
- ❌ バグを修正する → リファクタリングではない
- ❌ 新機能を追加する → リファクタリングではない
- ❌ 全部書き直す → リライト/リライト

### リライト(Rewrite)との違い

- **リファクタリング**: 少しずつ改善、動作は変えない、テストも通り続ける
- **リライト**: ゼロから書き直し、動作が変わることもある

### 現場での注意点

- 「リファクタリング」と言いながら機能追加すると、バグの原因がわからなくなる
- リファクタリング中もテストは常にグリーン（成功）を保つべき
- 大規模な書き直しを「リファクタリング」と呼ぶのは誤用

## テクニカルデット - 技術的負債、具体的には?

### 雰囲気使用例

```
「このコード、テクニカルデットだから後で直そう」
（単に「汚いコード」と言いたいだけ）
```

### 正しい使用例

```
「今は仮実装で進めます。これはテクニカルデットとして
記録し、次スプリントで返済計画を立てましょう」
```

### 何が違う?

テクニカルデット（技術的負債）は「借金」のメタファーです。

**借金の構造:**

- **元本**: 雑な実装で節約した時間
- **利息**: 保守や機能追加が遅くなるコスト
- **返済**: リファクタリングやテスト追加

重要なのは:

- **意図的に作る負債もある**（スピード優先の判断）
- **無自覚な負債もある**（単なる技術不足）
- 負債は放置すると利息がかさむ

### 現場での注意点

- 「テクニカルデット」は言い訳ではなく、ビジネス判断
- 「いつ返済するか」まで考えて初めて借金（デット）と言える
- 返済計画のない負債は、ただの「質の低いコード」

### ビジネスサイドへの説明例

```
「今回は仮実装で早く出します（借金）。
ただし次の機能追加時、倍の時間がかかります（利息）。
3スプリント後にリファクタリングが必要です（返済）」
```

## ポストモーテム - 死後?振り返り?

### 雰囲気使用例

```
「障害のポストモーテムやります」
（単なる「反省会」だと思ってる）
```

### 正しい使用例

```
「今回の障害のポストモーテムを作成しました。
根本原因と再発防止策を記載しています」
```

### 何が違う?

ポストモーテムは単なる振り返りではなく:
**「失敗から学び、再発を防ぐための構造化されたレポート」**

典型的な構成:

1. **何が起きたか** (事象)
2. **いつ起きたか** (タイムライン)
3. **なぜ起きたか** (根本原因)
4. **どう対応したか** (応急処置)
5. **今後どうするか** (再発防止策)

### 重要な文化

- **Blameless(非難しない)**: 個人を責めない
- **学びに焦点**: 「誰のせい」より「何を学んだか」
- **透明性**: 全員が見れる場所に公開

### 語源と背景

"Post-mortem"はラテン語で「死後の」という意味。医学の「死後解剖(検死)」から来ています。システムの「死」（障害）を解剖して原因を突き止める、というイメージです。

### 現場での注意点

- 単なる「反省会」とは違う。ドキュメントとして残すことが重要
- 犯人探しをしない。「プロセスの問題」として扱う
- Googleなどの企業ではポストモーテム文化が根付いている

## インクリメント - 増分?成果物?

### 雰囲気使用例

```
「今スプリントのインクリメント、これでいいですか?」
（「今週の成果物」くらいの意味で使ってる）
```

### 正しい使用例

```
「今スプリントのインクリメントは、前回までの機能に
ユーザー認証機能を追加したもので、実際にデプロイ可能です」
```

### 何が違う?

インクリメントは:
**「過去の全ての成果 + 今回の成果」を合わせた、動作するプロダクト**

つまり:

- ❌ 「今回作った機能だけ」ではない
- ✅ 「これまでの積み重ね + 今回の追加」
- ✅ すぐにリリース可能な状態
- ✅ テスト済みで動作する

### 例

- スプリント1: ログイン機能 → インクリメント: ログインできるアプリ
- スプリント2: プロフィール編集 → インクリメント: ログイン+プロフィール編集ができるアプリ
- スプリント3: パスワード変更 → インクリメント: 3つの機能が全部入ったアプリ

### 語源と背景

"Increment"は「増分」という意味。数学や物理で「少しずつ増える量」を指します。アジャイル開発では「機能を少しずつ積み重ねて製品を大きくする」という考え方です。

### 現場での注意点

- インクリメントは「動くソフトウェア」であることが前提
- 「コード書いたけどまだ動かない」はインクリメントではない
- スプリントごとに必ずインクリメントを作るのがスクラムのルール

## ベロシティ - 速度を測る意味とは

### 雰囲気使用例

```
「うちのチーム、ベロシティ高いから優秀だよね!」
（速度 = 優秀だと思ってる）
```

### 正しい使用例

```
「過去3スプリントの平均ベロシティが30ポイントなので、
次スプリントも30ポイント分の作業を計画しましょう」
```

### 何が違う?

ベロシティは:
**「チームが1スプリントで完了できる作業量の指標」**

- 単位は「ストーリーポイント」（作業の大きさの見積もり単位）
- チームごとに異なるので、他チームと比較できない
- 主な用途は「計画精度の向上」

### よくある誤解

- ❌ ベロシティが高い = 優秀なチーム
- ❌ ベロシティを上げることが目標
- ✅ ベロシティは「予測のための指標」
- ✅ 安定したベロシティが理想

### 計算例

```
スプリント1: 8ポイント完了
スプリント2: 5ポイント完了
スプリント3: 7ポイント完了
→ 平均ベロシティ = (8+5+7)/3 = 6.7ポイント
→ 次スプリントは6〜7ポイント分の作業を計画
```

### 現場での注意点

- ベロシティを「目標」にするとゲーム化される（見積もりを甘くするなど）
- チームAのベロシティ30とチームBのベロシティ30は全く別物
- 「今回ベロシティ上げよう!」は誤った目標設定

## コラム: 「先輩の使い方、実は間違ってた」事例集

### 事例1: スプリントという名の「ただの週」

**間違った使い方:**

```
月曜: 「今週のスプリント、何やる?」
水曜: 「やっぱりあっちの機能優先で」
金曜: 「来週に持ち越しで」
```

**何が問題:**
スプリントは途中で目標を変えないのが原則。これでは単なる「週」と変わりません。

---

### 事例2: バックログという名の「やることリスト」

**間違った使い方:**

```
バックログに全部詰め込んで、上から順にやっていく
→ 優先度の見直しなし
```

**何が問題:**
バックログは「優先度順」かつ「定期的に見直す」ことが重要。固定されたTODOリストではありません。

---

### 事例3: リファクタリングという名の「大改造」

**間違った使い方:**

```
「この機能、リファクタリングしますね」
→ 3日後、まだテストが通らない
→ 1週間後、仕様も少し変わってる
```

**何が問題:**
リファクタリングは「動作を変えない」が大原則。テストが通らない時点でリファクタリングではありません。

---

### 事例4: テクニカルデットという名の「言い訳」

**間違った使い方:**

```
レビュアー: 「このコード、ちょっと読みづらいですね」
開発者: 「テクニカルデットってことで...」
```

**何が問題:**
テクニカルデットは「意図的な判断」か「返済計画がある」場合に使う言葉。単なる低品質の言い訳ではありません。

## 章末チェック: あなたの理解度は?

次の文章の誤りを指摘できますか?

1. 「今スプリントは忙しいから、期間を1週間延長しよう」
   - ヒント: スプリント期間は...?

2. 「バックログを全部消化するまで頑張ろう!」
   - ヒント: バックログの性質は...?

3. 「リファクタリングついでに、この新機能も入れちゃいましょう」
   - ヒント: リファクタリングの定義は...?

4. 「テクニカルデットがあるから、このコードは汚い」
   - ヒント: 負債と単なる低品質の違いは...?

5. 「今スプリントのベロシティ、先月より高くしよう!」
   - ヒント: ベロシティの目的は...?

### 解答

1. **誤り**: スプリント期間は固定です。延長するのはスプリントの原則違反。途中で終わらないなら、次スプリントに回すべき。

2. **誤り**: バックログは「全部やる」ものではありません。優先度が低いものは永遠にやらないこともある。それでOK。

3. **誤り**: リファクタリングは「外部から見た動作を変えない」が大原則。新機能追加は明確に違う作業です。

4. **誤り**: テクニカルデットは「意図的な借金」または「返済計画がある負債」。単に質が低いだけのコードは負債とは呼べません。

5. **誤り**: ベロシティは「予測のための指標」であり、目標ではありません。無理に上げようとすると見積もりが歪みます。

## 第2章のまとめ

開発フロー用語は「なんとなく」使いがちですが、実は一つ一つに明確な定義があります。特にスクラム用語は、単なる言葉の置き換えではなく、背景にある「考え方」を理解することが重要です。

**キーポイント:**

- **スプリント**: 固定期間で完成品を作る約束
- **バックログ**: 優先度付きで常に見直される生きたリスト
- **リファクタリング**: 動作を変えずに内部改善
- **テクニカルデット**: 意図的な借金、返済計画が重要
- **ポストモーテム**: 非難せず学びを得る構造化レポート
- **インクリメント**: 過去+今回の全てを含む動作するプロダクト
- **ベロシティ**: 予測のための指標、目標ではない

次の第3章では、エンジニアが最も苦手とする「ビジネス会議で飛び交う用語」に挑みます!
